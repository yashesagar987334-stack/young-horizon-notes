# 个人网站项目需求说明

## 1. 项目定位

本项目旨在创建一个简洁、高级、以生活分享与记录为主的个人网站，用于集中展示以下内容：

- 生活照片 / 风景照片 / 日常记录
- 个人推文 / 随笔 / 想法记录 / 生活片段
- 指向本人其他作品与平台的链接
  - 例如：个人小网站、实验项目、小红书主页等

网站整体希望具备以下特征：

- 视觉上简洁、克制、现代
- 排版上精致、高级，尤其是“推文”部分要具有类似微信公众号文章页面的优质阅读体验
- 内容组织清晰，方便后续持续更新和分类管理
- 手机端和电脑端都可以直接访问，并保持良好的浏览体验
- 技术实现应尽量清楚、稳定、可维护，便于后续由非原作者继续修改和扩展

---

## 2. 我的角色设定

在本项目中，我将以**高级编程人员 / 高级前端工程师**的身份参与设计与开发。

我的工作原则如下：

- 以长期可维护性为优先，而不是只追求短期快速堆砌页面
- 用清晰的目录结构和职责划分，降低后续修改成本
- 保持代码风格统一，减少“看不懂、不敢改”的情况
- 尽量采用容易理解的实现方式，避免不必要的复杂度
- 为后续新增照片、推文、外部链接预留简单直接的扩展方式

这意味着：后续代码方案会优先考虑“你以后能不能方便维护”，而不仅仅是“现在能不能跑起来”。

## 3. 核心需求

### 3.1 内容展示需求

网站需要支持以下几类核心内容：

#### A. 生活照片 / 风景照片

- 展示日常拍摄内容，例如风景、旅行途中、生活瞬间等
- 形式可以是精选封面 + 照片列表 / 相册流
- 风格应偏简洁，避免过于拥挤
- 后续应方便继续新增图片

#### B. 推文 / 文字内容

- 用于发布短文、思考、记录、分享
- 页面排版参考微信公众号文章的阅读体验
- 要求有更好的标题层级、段落节奏、图片穿插、引用样式、留白控制
- 阅读体验应明显优于普通博客的默认排版

#### C. 推文分类管理

- 推文需要支持按主题分类管理
- 分类形式希望接近“文件夹”或 Notion 中多层内容整理的感觉
- 初步分类可以包括：
  - 旅游
  - 学习
  - 生活
  - 随想
- 后续应支持自由新增分类，而不需要改动复杂代码
- 列表页和详情页都应体现分类信息，方便浏览和归档

#### D. 外部链接聚合

- 展示并跳转到我自己的其他站点、项目、账号主页
- 可包含：
  - 小网站
  - 个人项目
  - 小红书
  - 其他社交平台或作品页
- 链接区域应整洁、有辨识度，不要做成廉价的“链接堆砌”

## 4. 页面目标

初步建议网站至少包含以下页面或内容模块：

### 4.1 首页

首页用于建立整体浏览入口，建议包含：

- 简短的站点介绍
- 视觉封面区域（可以是一张风景图、生活场景图，或具有氛围感的视觉区域）
- 进入推文区、照片区、链接区的导航入口
- 进入分类内容区的导航入口
- 精选内容预览

目标：

- 首页不需要信息过多，但要体现“高级、克制、有审美”
- 首页重点是内容入口和整体氛围，不强调强烈的个人形象展示

### 4.2 推文 / 文章列表页

- 展示所有推文或文字记录
- 列表样式应清晰，支持按时间浏览
- 支持按分类浏览，例如旅游、学习、生活等
- 分类切换应足够直观，最好具有接近“文件夹浏览”的整理感
- 每条内容应可进入详情页

### 4.3 推文 / 文章详情页

这是整个网站的重点页面。

目标风格：

- 阅读体验参考高质量微信公众号文章
- 重视字体、行高、段落间距、引用、图片说明、分隔方式
- 页面要有足够留白，突出内容本身
- 视觉上简洁，但不能显得普通

### 4.4 照片展示页

- 用于集中展示个人照片
- 照片内容以生活记录、旅行记录、风景记录为主
- 可采用瀑布流、网格、精选相册等形式
- 要保证手机和电脑上浏览都舒服

### 4.5 链接 / 足迹页

- 展示外部链接与个人数字名片
- 可以做成“我在这些地方”的形式
- 除了链接本身，最好带有简短说明，告诉访问者这个链接是做什么的

## 5. 设计方向

整体视觉关键词：

- 简洁
- 高端
- 克制
- 现代
- 有内容感

设计上应避免：

- 过度花哨的动画
- 过于廉价的卡片堆叠风格
- 大量无意义装饰
- 低质量默认字体与默认排版

设计上建议突出：

- 高质量留白
- 稳定的版心宽度与阅读节奏
- 有辨识度的字体组合
- 精致的正文排版
- 轻量但有质感的交互反馈

## 6. 技术实现原则

为了确保你后续可以自己维护，这个项目在技术实现上应遵守以下原则：

### 6.1 结构清晰

- 页面、组件、数据、样式分层明确
- 不把所有逻辑堆在一个文件里
- 文件命名统一、语义清楚

### 6.2 易于扩展

- 新增一篇推文时，不需要改很多地方
- 新增一组照片时，流程简单
- 新增一个外部链接时，最好只需要修改一处数据配置

### 6.3 易于阅读

- 避免炫技式代码
- 组件职责单一
- 关键逻辑尽量直观
- 复杂部分应有少量必要注释

### 6.4 样式可控

- 统一颜色、字体、间距、圆角、阴影等设计变量
- 尽量建立基础设计 token，避免后续样式越改越乱
- 推文排版样式应单独整理，避免和普通页面样式混在一起

### 6.5 内容与表现分离

- 推文内容、照片数据、外部链接数据尽量独立存放
- 页面负责渲染，数据负责内容
- 这样后续你维护时，不需要频繁进入复杂组件内部改代码

### 6.6 多端兼容

- 必须兼容手机端与桌面端访问
- 页面布局需要根据屏幕宽度自适应调整
- 推文正文在手机上也要保持良好的阅读体验，不能因为缩放而影响排版质量
- 照片、导航、分类切换、链接区域都需要兼顾触屏操作体验

## 7. 推荐的内容管理思路

为了方便长期维护，建议采用“轻内容管理”方式，而不是一开始就接入很重的后台系统。

建议方向：

- 推文内容使用结构化文件管理
  - 例如 Markdown / MDX / JSON 等
- 照片信息通过独立数据文件管理
- 外部链接通过统一配置文件管理

这样做的好处：

- 改内容时不必碰复杂代码
- 版本管理清楚
- 后期若需要再升级为 CMS，也更容易迁移

## 14. 当前项目实现说明

当前仓库已经按以下方式初始化：

- 使用 Astro + TypeScript + MDX 作为站点基础
- 使用 `src/content/notes` 存放所有记录内容
- 使用 `src/data/categories.ts` 管理一级分类和二级分类
- 使用 `src/data/links.ts` 管理外部链接
- 使用 GitHub Actions 自动部署到 GitHub Pages

这意味着，后续你新增一篇内容时，默认**不需要修改页面代码**。

## 15. 后续如何新增文章

你以后新增文章时，按这个流程即可：

1. 把封面图或正文图片放到 `public/images/covers` 或 `public/images/notes`
2. 在 `src/content/notes` 新建一个 `.mdx` 文件
3. 复制 `src/content/notes/template.mdx` 作为模板
4. 修改标题、摘要、日期、分类、标签和正文
5. 提交到 GitHub 后，站点会自动更新

文章模板如下：

```mdx
---
title: ""
description: ""
date: "2026-04-21"
parentCategory: "travel"
childCategory: "city-walk"
tags: []
featured: false
draft: true
cover: "/images/covers/cover-travel.svg"
---

在这里开始写正文。
```

字段说明：

- `title`：文章标题
- `description`：列表页和详情页用到的摘要
- `date`：发布日期
- `parentCategory`：一级分类
- `childCategory`：二级分类
- `tags`：标签
- `featured`：是否出现在首页精选区
- `draft`：是否为草稿
- `cover`：封面图路径

说明：

- `draft: true` 时，文章不会显示在公开站点
- `draft: false` 后，文章会自动参与生成首页、分类页和详情页

## 16. 后续如何新增链接和分类

如果你要新增一个外部链接：

- 修改 `src/data/links.ts`

如果你要新增一个分类：

- 修改 `src/data/categories.ts`

只要内容文件中的分类字段和配置文件保持一致，页面就会自动归类显示。
